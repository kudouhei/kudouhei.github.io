flex + justify-content + align-items,只需设置父节点属性，无需设置子元素（IE11）

  <div class="child">Demo</div>
</div>

  .parent {
    display: flex;
    justify-content: center; /* 水平居中 */
    align-items: center; /*垂直居中*/
  }
</style>

2. table水平垂直居中
inline-block + text-align + table-cell +vertical-align 

<div class="parent">
  <div class="child">Demo</div>
</div>

<style>
  .parent {
    text-align: center;//水平居中
    display: table-cell;
    vertical-align: middle;//垂直居中
  }
  .child {
    display: inline-block;//防止块级元素宽度独占一行，内联元素可不设置
  }

MVVM原理

发表于 2018-09-04

MVVM原理

  • View层：代表视图、模板，负责将数据模型转化为UI展现出来
  • Model层：代表模型、数据，可以在Model层中定义数据修改和操作的业务逻辑
  • ViewModel层：连接Model和View

MVVM架构下，View层和Model层没有直接联系，而是通过ViewModel层进行交互。ViewModel层通过双向数据绑定将View层和Model层连接起来，所以开发者只需要关注业务逻辑，无需手动操作DOM。

css性能优化

发表于 2018-09-02

CSS性能优化

1.内联首屏关键css
CSS直接内联到HTML文档中能使CSS更快速地下载，内联css能够使浏览器开始页面渲染的时间提前，因为在HTML下载完成之后就能渲染了。
由于初始拥塞窗口存在限制（TCP相关概念，通常是 14.6kB，压缩后大小），如果内联CSS后的文件超出了这一限制，系统就需要在服务器和浏览器之间进行更多次的往返，这样并不能提前页面渲染时间。

2.异步加载CSS
CSS会阻塞渲染，在CSS文件请求、下载、解析完成之前，浏览器将不会渲染任何已处理的内容，直到CSSOM构建完毕。

  • 使用JavaScript动态创建样式表link元素，并插入到DOM中。

  • 将link元素的media属性设置为用户浏览器不匹配的媒体类型（或媒体查询），如media=”print”，甚至可以是完全不存在的类型media=”noexist”。对浏览器来说，如果样式表不适用于当前媒体类型，其优先级会被放低，会在不阻塞页面渲染的情况下再进行下载。

    <link rel="stylesheet" href="mystyle.css" media="noexist" onload="this.media='all'">

  • rel="preload"异步加载资源,as是必须的。

    <link rel="preload" href="mystyle.css" as="style" onload="this.rel='stylesheet'">

3.文件压缩
4.去除无用CSS
一种是不同元素或者其他情况下的重复代码（提取公共类，减少重复）；一种是整个页面内没有生效的css代码。

有选择的使用选择器
css选择器的匹配是从右向左进行的。不使用嵌套过多过于复杂的选择器。

  • 为什么css选择器要从右往左解析
    • 如果正向解析，如[div p em]，首先要检查当前元素到html的整条路径，找到最上层的div，在往下找，如果遇到不匹配就必须回到最上层那个div，往下再去匹配选择器中的第一个div，回溯若干次才能确定匹配与否，效率很低。
    • 逆向匹配不同，如果当前的DOM元素是div，而不是selector最后的em，只要一步就能排除。只有在匹配时，才会不断向上找父节点进行验证。

5.优化重排与重绘

  • 重排：渲染层内的元素布局发生修改，都会导致页面重新排列，比如窗口的尺寸发生变化、删除或添加DOM元素，修改了影响元素盒子大小的CSS属性。获取某些属性，浏览器为了获得正确的值也会触发重排。

    • offset
    • scroll
    • client(top/left/width/height)
    • width,height
    • 调用了getComputedStyle()或者IE的currentStyle

  • 重绘：绘制，即渲染上色，所有对元素的视觉表现属性的修改，会引发重绘

服务端渲染

发表于 2018-08-31

服务端渲染SSR

单页应用需要等JS加载完毕后在前端渲染页面，就是说在js加载完毕并开始执行渲染操作前的这段时间里浏览器会产生白屏。

服务端渲染（Server Side Render，简称SSR）的意义在于弥补主要内容在前端渲染的成本，减少白屏时间，提升首次有效绘制的速度。可以使用服务端渲染来获得更快的首次有效绘制。

webpack配置重点

path模块
path是Node.js中的一个模块，用于处理目录的对象。

常用方法：

  • path.join():用于连接路径
  • path.resolve():用于将相对路径转为绝对路径

process
process对象是Node的一个全局对象，提供当前Node进程的信息。 

process对象提供一系列属性，用于返回系统信息  
process.argv： 返回当前进程的命令行参数数组  
process.env： 返回一个对象，成员为当前Shell的环境变量，比如process.env.HOME  
process.pid： 当前进程的进程号

webpack-merge

  • 在开发环境中，我们需要具有强大的、具有实时重新加载(live reloading)或热模块替换(hot module replacement)能力的 source map 和 localhost server。
  • 在生产环境中，我们的目标则转向于关注更小的 bundle，更轻量的 source map，以及更优化的资源，以改善加载时间。
  • 由于要遵循逻辑分离，我们通常建议为每个环境编写彼此独立的 webpack 配置。通用的配置部分，我们抽象出一个公共文件，通过 webpack-merge 工具的“通用”配置，我们不必在环境特定的配置中重复代码。

ExtractTextWebpackPlugin

ExtractTextWebpackPlugin 插件通常用来做样式文件的分离，被分离的文件不会被内嵌到 JS bundle 中，而会被放到一个单独的文件中，在样式文件比较大的时候，能够提前样式的加载。

html-webpack-plugin
通常在开发中，我们为了避免 CDN 和浏览器的缓存通常会个输出文件 bundle.js 加上一个hash 值例如 [hash].bundle.js，使用 html-webpack-plugin 能够在创建新的 html 文件的时候将我们把带有哈希值的 bundle.js 引用到 html 文件.

webpack的loader和plugin区别

  • loader用于加载某些资源文件。因为webpack本身只能打包commonjs规范的JS文件，对于其他资源如css、图片或者jsx等没有办法加载。所以就需要对应的loader将资源转化，加载进来。loader用于加载，作用于一个文件上。
  • plugin用于扩展webpack功能，而loader只专注于转化文件，plugin功能更丰富

babel工作原理

babel的主要功能就是个编译器，输入语言ES6+，编译目标语言是ES5。以字符串的形式将源代码传给他，它就会返回一段新的代码字符串。

babel的编译过程

  • 解析：使用Babylon解析器，会根据输入的js代码将代码字符串解析成抽象语法树
  • 变换：对抽象语法树进行变换操作
  • 再建：使用babel-generator根据变换后的抽象语法树再生成代码字符串

Vuex

主要用于对复杂应用进行状态管理。集中式存储管理应用的所有组件的状态，并以相应的规则保证状态以一种可预测的方式发生变化。 

- 状态：泛指vue组件的一些当前的状况和性质
- 状态管理：对状态进行管理，如a组件这个状态需要通知b组件。（state会显示到View，用户会根据View上的内容进行操作，触发actions，然后再去影响state。）

vuex流程

  • 每次vue组件需要给vuex分派一个actions，actions提交一个mutation，由mutation来修改state，然后再返回给vue组件渲染。
  • state状态只能由mutation来修改
  • actions可以封装各自mutation来进行修改state
  • 总的来说，vuex的状态是存储在store里面的，要修改状态就只能进行mutation的提交commit，一旦进行提交成功，状态被改变后，相关使用该状态的组件也会更新状态信息。

使用vuex的时候怎么用？ 

import Vue from 'vue';
import Vuex from 'vuex';
Vue.use(Vuex);
new Vue.Store({
    state: {  // 放置state的值
        count: 0,
        str: "abcd234"
    },
    getters: {  // 放置getters方法
        strLen: state => state.str.length        
    },
    // mutations只能是同步操作
    mutations: {  // 放置mutations方法
        increment(state, payload){
            state.count = payload.number
        }
    },
    // actions可以是异步操作
    actions: {  // 放置actions方法
        actionName({commit}){
            commit('mutationName')
        }        
    }
});
new Vue({
    el: '#app',
    store,
    ...
})

console.log(Vuex)可以看到，import进来的Vuex实际上是一个对象，里面包含了store构造函数，还有mapActions、mapGetters、mapMutations、mapState辅助方法。

创建好的store实例怎么在各个组件中都能引用

因为store实例是个全局单例，Vuex通过store选项，提供了一种机制将状态从根组件‘注入’到每一个子组件中。

通过在根实例中注册store选项，该store实例会注入到根组件下的所有子组件中，且子组件能通过this.$store访问到。

State
组件所发生的变化，也是直接受到State的驱动来变化的。

通过this.$store.state来直接获取状态，也可以利用vuex提供的mapState辅助函数将state映射到计算属性computed中。

Getters
本质上是用来对状态进行加工处理。getter的返回值会根据它的依赖被缓存起来，且只有当它的依赖值发生了改变才会被重新计算。通过this.$store.getters.valueName对派生出来的状态进行访问。

Mutations
利用mutations可以改变状态。更改vuex的store中的状态的唯一方法就是提交(commit)mutation.但是mutation触发状态改变的方式有些特别，所谓commit一个mutation，实际就像触发一个事件一样，传入一个mutation的类型以及携带一些数据payload载荷。

可以使用this.$store.commit('xxx')提交mutation，还可以使用mapMutation辅助函数将组件中的methods映射为this.$store.commit。

Actions
Action可以包含异步操作 

actions: {
    increment (context) {
      context.commit('increment')
    }
  }

它接收的第一个参数是一个与 store 实例具有相同方法和属性的 context 对象，因此你可以调用 context.commit 提交一个 mutation，或者通过 context.state 和 context.getters 来获取 state 和 getters。

mutation处理函数中所做的事情是改变state，而action处理函数中所做的事情则是commit mutation。

触发action，使用this.$store.dispatch(actionType, payload)

流程上
vue component—-dispatch—->actions—-commit—->mutations—-mutate—->state—-render—->vue component

JavaScript定义类的三种方法

发表于 2018-08-02

JavaScript定义类的三种方法

1. 构造函数法
用构造函数模拟“类”，在其内部用this关键字指代实例对象。用new生成实例。类的属性和方法还可以定义在构造函数的prototype对象上。 

funciton Foo(){
    this.name = "Foo";
    this.getName = function(){}
}
var foo = new Foo();
console.log(foo.name); // Foo
Foo.prototype.getName = function(){}

优点：可以根据参数来构造不同的对象实例
缺点：构造时每个实例对象都会生成getName方法，造成内存浪费

2. 原型方式 

function Foo(){}
Foo.prototype.name = "hei";
Foo.prototype.getName = function(){}
var f1= new Foo()

原型方式缺点不能通过参数来构造对象实例，优点是所有对象实例都共享getName方法。

3. 组合方式 

function Foo(name){
    this.name = name;
}
Foo.prototype.getName = function(){
    return this.name;
}

2. Object.create()
直接用Object.create()生成实例，不需要用到new

var Foo = {
    name: "Foo",
    getName: function(){}
}
var foo = Object.create(Foo);

兼容老式浏览器 

if(!Object.create){
    Object.create = function(o){
        function F(){};
        F.prototype = o;
        return new F();
    }
}

此方法不能实现私有属性和私有方法，实例对象之间也不能共享数据，对“类”的模拟不够全面。

3. 封装（推荐）

3.1 封装

这种方法不使用this和prototype，代码简单，用对象模拟“类”。在这个类里面，定义一个构造函数，用来生成实例。在构造函数里，定义一个实例对象，把这个实例对象作为返回值。 

var Foo = {
    createFoo: function(){
        var foo = {};
        foo.name = "foo";
        foo.getName = function(){};
        return foo;
    }
}

使用的时候，调用createFoo()方法，就可以得到实例对象。

var h1 = Foo.createFoo();
h1.getName();

此方法好处是，结构清晰。

3.2 继承

让一个类继承另一个类，只需要调用createFoo()方法即可。

}

定时器问题
我在a页面写一个定时，让他每秒钟打印一个1，然后跳转到b页面，此时可以看到，定时器依然在执行。这样是非常消耗性能的。

方法1：
首先在data函数里面进行定义定时器名称： 

data(){
    return {
        timer: null
    }
}

使用定时器： 

this.timer = (() => {
    // 某些动作
}, 1000)

最后在beforeDestroy()生命周期内清除定时器 

beforeDestroy() {
    clearInterval(this.timer);
    this.timer = null;
}

方案2：
通过$once这个事件侦听器在定义完定时器之后的位置来清除定时器。 

const timer = setInterval(() => {
    // 某些定时器操作
}, 500)
// 通过$once来监听定时器，在beforeDestroy钩子可以被清除
this.$once('hookl:beforeDestroy', ()=>{
    clearInterval(timer)
})

vue keep-alive实践

在vue项目中有某些组件不需要多次渲染，需要将组件在内存中进行缓存，这时候应用keep-alive。它可以使被包含的组件状态维持不变，即便组件切换了，其内的状态依旧维持在内存中，下一次显示，不会重新渲染。

<keep-alive>
<component>
  <!-- 组件将被缓存 -->
</component>
</keep-alive>

配合router-view使用
有时候需要将整个路由页面一起缓存下来，就是将<router-view>缓存

<keep-alive>
    <router-view v-if="$route.meta.keepAlive"></router-view>
</keep-alive>
<router-view v-if="!$route.meta.keepAlive"></router-view>

new Router({
    routes: [
        {
            name: 'a',
            path: '/a',
            component: A,
            meta: {
                keepAlive: true
            }
        },
        {
            name: 'b',
            path: '/b',
            component: B
        }
    ]
})

a路由便达到了被缓存的效果，如果还有想要缓存的路由，只需要在路由元中加入 keepAlive: true.

使用router.meta
需求： 默认显示A；B跳转到A，A不刷新；C跳到A，A不刷新

// 在A路由里面设置meta属性
{
    path: '/',
    name: 'A',
    component: A,
    meta: { keepAlive: true // 需要被缓存}
}
// 在B组件里面设置beforeRouteLeave
export default {
    data() {
        return {};
    },
    methods: {},
    beforeRouteLeave(to, from, next){
        // 设置下一个路由的meta
        to.meta.keepAlive = true; // 让A缓存，即不刷新
    }
}

JavaScript事件

事件级别

  • DOM0：element.onclick = function(){}
  • DOM2: element.addEventListener(‘click’,function(){}, false)
  • DOM3:element.addEventListener(‘keyup’,function(){},false)

事件流(事件三个阶段)
事件流三个阶段，首先捕获阶段，事件通过捕获，到达目标阶段，从目标元素在上传到window对象，为冒泡阶段

描述DOM事件捕获的具体流程
window -> document -> html标签(document.documentElement) -> body ->…->目标元素

Event对象

  • event.preventDefault()阻止默认行为
  • event.stopPropagation()阻止冒泡
  • event.stopImmediatePropagation()注册两个事件，优先级执行
  • event.currentTarget当前绑定事件的对象
  • event.target：触发事件的某个具体的对象，只会出现在事件机制的目标阶段

先捕获，再冒泡
在target注册的监听器，不分捕获和冒泡

事件委托
事件委托是利用事件的冒泡原理来实现的，何为事件冒泡呢？就是事件从最深的节点开始，然后逐步向上传播事件

function delegateEvent(ele, selector, type, fn){
    if(ele.addEventListener){
        ele.addEventListener(type, eventfn, false);
    }else if(ele.attachEvent){
        ele.attachEvent("on"+type, eventfn);
    }else{
        ele['on'+type] = eventfn;
    }
    function eventfn(e){
        var e = e||window.e;
        var target = e.target || e.srcElement;
        if(target.matches(selector)){
            if(fn){
                fn.call(target, e);
            }
        }
    }
}
