对于单行省略

对于多行省略
WebKit内核的浏览器实现起来比较简单，可以通过添加一个-webkit-line-clamp的私有属性来实现，-webkit-line-clamp是用来限制在一个块元素显示的文本的行数。

overflow : hidden;
text-overflow: ellipsis;
display: -webkit-box;
-webkit-line-clamp: 2;
-webkit-box-orient: vertical;

其他浏览器，利用相对定位在最后面加上一个省略号。 

<p>XX</p>
p{
    position: relative;
    line-height:1.5em;
    /*高度为要显示的行数x行高*/
    height: 3em;
    overflow: hidden;
}
p:after{
    content: '...';
    position: absolute;
    bottom: 0;
    right:0;
    padding:0 5px;
    background-color: #fff;
}

行内元素视觉格式化

  • 非替换元素：将内容直接告诉浏览器，将其显示出来，例如包含文字的段落
  • 替换元素：浏览器根据其标签的元素与属性来判断显示的具体内容。例如img、textarea、input
    • 这些元素拥有内在尺寸，可以设置width/height属性
    • 内容区高度值=padding-top + padding-bottom + margin-top + margin-bottom + height。
  • 行间距：行间距为line-height和font-size值之差。行间距只应用于非替换元素，可为负。
  • 行内框：行内框是通过向内容区增加行间距来描述。对于非替换元素，元素行内框的高度刚好等于line-height值。对于替换元素，元素行内框的高度等于内容区高度。

offsetWidth、clientWidth、scrollWidth如何计算

  • offsetWidth,offsetHeight:包含所有边框的可视框的大小。可以通过添加宽高和padding、边框。
  • clientWidth,clientHeight:框内容的视觉部分，不包括边框或滚动条，但包括填充。
  • scrollWidth，scrollHeight：所有框的内容大小，包括当前在滚动区域之外隐藏的部分。

css实现自适应正方形

1. CSS3 vw单位
vw是相对于视口宽度百分比的单位，1vw = 1% viewport width

<div class="placeholder"></div>
.placeholder{
  width:50vw;
  height:50vw;
}

2. 设置垂直方向的padding撑开容器
margin, padding 的百分比数值是相对父元素宽度的宽度计算的。由此可以发现只需将元素垂直方向的一个 padding 值设定为与 width 相同的百分比就可以制作出自适应正方形。 

.placeholder{
  width:100%;
  padding:bottom:100%;
}

缺点：如果正方形中有内容，容器的高度就会被拉伸，可以通过设置height：0解决。

3. 利用伪元素的margin(padding)-top撑开容器
利用百分比数值的 padding-bottom 属性撑开容器内部空间，但是这样做会导致在元素上设置的 max-height 属性失效。失效的原因是max-height 属性只限制于 height，也就是只会对元素的 content height 起作用。

.placeholder {
  width: 100%;
  overflow: hidden;
}
.placeholder:after {
  content: '';
  display: block;
  margin-top: 100%; /* margin 百分比相对父元素宽度计算 */
}

列表数据展示优化

某用户有 1000 条交易数据需要在客户端全部展示，那么，这种场景下，该怎么去优化列表数据的展示呢？

按需显示
按需显示是指根据容器元素的高度以及列表项元素的高度来显示数据。 

const demandData = originalData.slice(0, Math.ceil(containerHeight / itemHeight))

originalData 表示原始数据，containerHeight 表示容器元素高度，itemHeight 表示列表项高度，demandData 表示按需显示需要的数据，scrollTop 表示容器元素的 Y 轴滚动值。

当页面发生滚动时，重新计算容器元素内所能展示的第一个数据项和最后一个数据项，然后根据originalData进行对应的数据项截取。 

const from = Math.floor(scrollTop / itemHeight);
const to = Math.ceil((scrollTop + containerHeight) / itemHeight);

const demandData = originalData.slice(from, to)

延迟显示
延迟加载是一种常见的首屏优化方案，其基本思路是应用初始化时，首屏根据客户端的 Viewport 只加载用户可见的部分，当用户滚动对应位置时，才去加载对应元素，以缩减首屏的加载时间。

列表初始化时，根据容器元素的高度以及列表项元素的高度，依然只是加载列表初始化所需的数据项。

const demandData = originalData.slice(0, Math.ceil(containerHeight / itemHeight))

当用户往下滚动时，再动态计算需要往列表后追加的数据项。 

const from = demandData.length; 
const to = Math.ceil((scrollTop + containerHeight) / itemHeight);    
const demandData = [].concat(demandData, originalData.slice(from, to))

与按需显示不同的是，延迟显示不再是对 originalData 进行截取，而是对 demandData 进行追加，直到用户滚动到底部，所有数据全部显示完毕为止。

页面上有个空的无序列表节点，用 <ul></ul> 表示，要往列表中插入 30000个 <li>，每个列表项的文本内容是列表项的插入顺序，取值 1, 2, 3, ... ,30000，怎么用原生的 JS 实现这个需求？

思路：可以从减少 DOM 操作次数、缩短循环时间两个方面减少主线程阻塞的时间。减少 DOM 操作次数的良方是 DocumentFragment；而缩短循环时间则需要考虑使用分治的思想把 30000 个

  • 分批次插入到页面中，每次插入的时机是在页面重新渲染之前。

    • (()=>{
    const ndContainer = document.getElementById('js-list');
    if(!ndContainer){
        return;
    }
    const total = 30000;
    const batchSize = 4 //每批插入的节点次数
    const batchCount = total/batchSize //需要批处理次数
    let batchDone = 0 //已经完成批处理个数
    function appendItems(){
        const fragement = document.createDocumentFragment();
        for(let i=0;i<batchSize;i++){
            const ndItem = document.createElement('li');
            ndItem.innerText = (batchDone * batchSize)+i+1;
            fragment.appendChild(ndItem);
        }
        ndContainer.appendChild(fragment);
        batchDone +=1;
        doBatchAppend();
    }
    function doBatchAppend(){
        if(batchDone < batchCount){
            window.requestAnimationFrame(appendItems);
        }
    }
    doBatchAppend();
    ndContainer.addEventListener('click',function(e){
        const target = e.target;
        if(target.tagName === 'LI'){
            alert(target.innerHTML);
        }
    })
})()

    前端性能优化

    1. 网络方面（网络传输性能）
    浏览器同一时间所发送的网络请求数是有限的。浏览器处理用户请求的过程：浏览器在得到用户请求之后，经历了下面阶段：重定向->拉取缓存->DNS查询->建立TCP链接->发起请求->接收响应->处理HTML元素->元素加载完成

    • 减少http请求，必要的资源合并
    • 减小资源体积，资源压缩(图片体积，雪碧图，使用字体图标，使用webp格式)
    • 缓存
      • DNS缓存：浏览器会在DNS解析步骤中消耗一定的时间
        • DNS预解析
          • <link rel="dns-prefetch" href="A.com">
          • <meta http-equiv="x-dns-prefetch-control" content="on">
        • 当浏览网页时，浏览器会在加载网页时对网页中的域名进行预解析并缓存，这样在浏览器加载网页中的链接时，就无需进行DNS解析，减少用户的等待时间，提高用户体验
      • CDN部署与缓存：CDN作为静态资源文件的分发网络，本身就已经提升了，网站静态资源的获取速度，加快网站的加载速度，同时也给静态资源做好缓存工作，有效的利用已缓存的静态资源，加快获取速度
      • http缓存：给资源设定缓存时间，防止在有效的缓存时间内对资源进行重复的下载，从而提升整体网页的加载速度

    浏览器默认的缓存是放在内存里，内存里的缓存会因为进程的结束或浏览器的关闭而被清除，而存在硬盘里的缓存才能够被长期保留下去。from memory cache和from dis cache，通过在服务器上设置的Etag字段控制缓存存放的位置，在浏览器接收到服务器响应后，会检测响应头部Header，如果有Etag字段，那么浏览器就会将本次缓存写入磁盘中。

    拉取缓存会出现200、304两种不同的状态码，取决于浏览器是否有向服务器发起验证请求。 只有向服务器发起验证请求并确认缓存未被更新，才会返回304状态码。

    2. 渲染和DOM操作方面

    • 减少重排和重绘
      • DOM渲染层与GPU硬件加速：布局是由CPU处理的，绘制则是由GPU完成的
        • 浏览器会先获取DOM树并依据样式将其分割成多个独立的渲染层
        • CPU将每个层绘制进绘图中
        • 将位图作为纹理上传至GPU绘制
        • GPU将所有的渲染层缓存，如果下次上传的渲染层没有发生变化，GPU就不需要对其进行重绘，并复合多个渲染层最终形成图像
      • 重排：渲染层内的元素布局发生修改，都会导致页面重新排列，比如窗口的尺寸发生变化、删除或添加DOM元素，修改了影响元素盒子大小的CSS属性。获取某些属性，浏览器为了获得正确的值也会触发重排。
        • offset
        • scroll
        • client(top/left/width/height)
        • width,height
        • 调用了getComputedStyle()或者IE的currentStyle
      • 重绘：绘制，即渲染上色，所有对元素的视觉表现属性的修改，会引发重绘
    • 优化网页渲染
      • css的文件放在头部，js文件放在尾部或者异步加载
        • 动态脚本加载
        • defer：defer是在HTML解析完之后才执行，如果是多个，按照加载的顺序依次执行
        • async：async是在加载完之后立即执行，如果是多个，执行顺序和加载顺序无关
      • 尽量避免内联样式
    • DOM操作优化
      • 避免在document上直接进行频繁的DOM操作
      • 使用classname代替大量的内联样式修改
      • 使用requestAnimationFrame代替setInterval操作
      • 使用事件代理

    浏览器显示页面的原理

    流程

    1. 获取HTML文档及样式表文件
    2. 解析成对应的树形数据结构
      • DOM tree
      • CSSOM tree
    3. 计算可见节点形成render tree
    4. 计算DOM的形状及位置进行布局
    5. 将每个节点转化为实际像素绘制到视口上

    资源加载：

    • 执行JavaScript会阻止DOM tree构建
    • 加载CSS会阻止render tree构建

    默认情况下，JavaScript脚本会在引入它的位置执行，这是会阻断DOM tree的构建；如果在运行脚本时浏览器尚未完成CSS的下载和CSSOM tree的构建，浏览器会将脚本执行延迟到这些操作结束之后。

    CSS3主要新特性

    盒模型

    • 可以通过设置盒模型属性box-sizing:border-box（默认值：content-box）来让W3C兼容传统的IE盒模型。
    • box-shadow增加阴影
    • 边框圆角：border-radius

    CSS3选择器

    • E[foo^="bar"]，选择匹配E的元素，且该元素定义了foo属性，foo属性值以“bar”开始。
    • E[foo$="bar"],foo属性值以“bar”结束
    • E[foo*="bar"],foo属性值包含“bar”
    • :first-child,选中的元素集合中的第一个元素
    • :not,否定选择器
    • :nth-child,选择器匹配属于其父元素的第N个子元素，不论元素的类型。如果第N个子元素与选择的元素类型不同则样式无效！（div+空格，只根据父元素选择）
    • :nth-of-type,选择器匹配属于父元素的特定类型的第N个子元素的每个元素。

    CSS3变形：transform

    CSS3过渡：transition

    • 每组transition包含四个属性：transition-property，transition-duration，transition-time-function，transition-delay
    • transition是异步的

    CSS3动画：通过设置@keyframes来定义一种动画效果，动画的实现用animation。

    CSS3伸缩布局

    • display：flex|inline-flex。值为flex时将会按block元素进行渲染。
    • 伸缩容器属性
      • 伸缩主轴方向：flex-direction
      • 伸缩容器内项目是否换行：flex-wrap
      • 主轴对齐方式：justify-content
      • 侧轴对齐方式：align-items
    • 伸缩项目的属性
      • 显示顺序：order
      • 伸缩性放大缩小比例：flex：none|flex-grow|flex-shrink|flex-basis

    CSS3背景

    • background-size：规定背景图片的尺寸
    • background-origin：规定背景图片的定位区域(content-box|padding-box|border-box)
    • background-clip:规定背景的绘制区域，参数同上。

    移动端前端开发：分为移动端web app和移动端网页

    移动端网页布局和PC差异
    主要是css方面，外加如何做到同一url，不同客户端展现不一致的做法，俗称pc和mobile都兼容。

    rem在移动端布局

    移动端的字体默认为16px，那么1rem我想表示为10px的话，我们就需要 10 % 16 = 0.625 即为62.5%，这样就可以比较方便的把设计稿里的px转换成rem单位来做到自适应了。这样无论用户如何设置电脑或者手机的默认字体大小，设置为rem的单位的长度都会随比例变化。

    CSS盒模型

    • 标准盒模型
      • width/height指的是content区域的宽度和高度
      • 一个块的总宽度=width+margin+padding+border
    • IE盒模型
      • width/height指的是content+padding+border
      • 一个块的总宽度=width+margin

    如何设置这两种模型：
    box-sizing: content-box(默认)，border-box(IE模型)

    js如何设置获取盒模型对应的宽高

    1. dom.style.width/height只能取内联样式的宽和高
    2. dom.currentStyle.width/height只能IE浏览器
    3. window.getComputedStyle(dom).width/height通用性最好
      • getComputedStyle会获取当前元素所有最终使用的css属性值，以数组形式返回，只读。
      • getComputedStyle会引起回流，因为它需要获取祖先节点的一些信息进行计算(譬如宽高等)。其他一些会引发回流的操作，如offsetXXX，scrollXXX，clientXXX，currentStyle等
    4. dom.getBoundingClientRect().width/height返回元素的大小及其相对于视口的位置。

    BFC

    块级格式化上下文，BFC是一个独立的布局环境，其中的元素布局是不受外界的影响，并且在一个BFC中，块盒与行盒都会垂直的沿着其父元素的边框排列。
    原理：

    • 垂直方向会发生边距重叠
    • BFC区域不会与浮动元素的box重叠，可以用来清除浮动
    • BFC在页面上是一个容器，外面的元素不会影响里面的元素
    • 计算BFC高度的时候，浮动元素也会参与计算

    如何创建一个BFC

    • float值不为none
    • position的值不是static或relative
    • display属性为inline-box，table相关
    • overflow为auto，hidden

    使用场景

    • 利用BFC避免外边距折叠
      • BFC可能会造成外边距折叠，也可以利用它来避免这种情况
      • BFC产生外边距折叠要满足一个条件：两个相邻元素要处于同一个BFC中。所有若两个相邻元素在不同的BFC中，就能避免外边距折叠。
    • BFC包含浮动
      • 浮动元素会脱离文档流，如果一个没有高度或者height为auto的容器的子元素是浮动元素，则该容器的高度是不会被撑开的。通常会利用伪元素(:after或:before)来解决这个问题
      • BFC能包含浮动，也能解决容器高度不会被撑开的问题
    • 使用BFC避免文字环绕
      • 在BFC上下文中，每个盒子的左外侧紧贴包含块的左侧，有浮动也是如此，除非盒子创建了一个新的BFC。
    • 在多列布局中使用BFC
      • 如果创建一个占满整个容器宽度的多列布局，在某些浏览器中最后一列有时会掉到下一行。可能是因为浏览器四舍五入了列宽从而所有列的总宽度会超出容器
      • 如果我们在多列布局中的最后一列里创建一个新的BFC，它将总是占据其他列先占位完毕后剩下的空间。
